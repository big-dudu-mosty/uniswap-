# Full-Stack DEX Project Constitution

> **项目北极星**：本文档定义不可协商的开发原则与质量铁律。任何技术方案设计前必须逐条审查，未通过必须给出充分理由。

---

## I. 核心设计哲学

### 1. 去中心化第一原则
- **私钥管理**：前端永远不存储、不传输用户私钥，所有交易必须通过 MetaMask 签名
- **后端定位**：后端服务仅提供只读数据查询与分析，禁止执行任何链上交易
- **用户自主权**：用户完全掌控自己的资产和私钥，DEX 不托管任何资金

### 2. 简单性优先原则
- 禁止引入不必要的抽象层和设计模式
- 优先使用标准库和成熟框架（Solidity: OpenZeppelin, Backend: NestJS, Frontend: React）
- 三行代码能解决的问题不要创建新的工具函数
- 不要为假想的未来需求设计功能

### 3. 安全性不可妥协原则
- **智能合约**：所有外部调用必须检查重入攻击、整数溢出、授权验证
- **前端**：所有用户输入必须验证，防止 XSS 和注入攻击
- **后端**：使用 class-validator 严格校验所有 API 输入
- **密钥管理**：环境变量管理敏感信息，禁止硬编码任何密钥或私钥
- **审计机制**：关键操作必须有完整的日志记录

## II. 代码质量铁律

### 1. 禁止引入全局可变状态
- 禁止使用全局变量（除配置常量外）
- 禁止滥用单例模式（NestJS 的依赖注入除外）
- 状态管理必须遵循框架规范：React 用 Zustand/Context，Backend 用 NestJS DI

### 2. 错误处理强制规范
- **所有 error 必须在当前作用域被处理**，不允许吞掉异常
- Solidity: 使用 `require()` / `revert()` 提供清晰的错误信息
- TypeScript: 使用 try-catch 或 Promise.catch()，记录详细错误上下文
- 前端：用户友好的错误提示 + Console 详细日志

### 3. 输入验证强制要求
- **所有对外接口必须有明确的输入校验**
- 智能合约：检查地址非零、数量大于零、授权有效
- 后端 API：使用 `class-validator` DTO 校验
- 前端：表单验证 + 合约调用前的前置检查

### 4. 重构行为等价原则
- **重构时必须保持行为等价，禁止"顺手优化"**
- 功能改进与重构必须分离为不同的提交
- 重构后必须通过原有测试用例

### 5. 变更范围控制原则
- **修改超过 3 个文件时，必须拆分为多个步骤**
- 每个 commit 应聚焦单一目的
- 大型重构必须先制定 plan.md，拆分成 tasks.md

### 6. 可读性优先原则
- **不允许为了"简洁"牺牲可读性**
- 命名要清晰表达意图，禁止缩写（除约定俗成外）
- 复杂逻辑必须有注释说明"为什么"而非"是什么"
- 避免过度嵌套（建议最多 3 层）

### 7. 测试覆盖强制要求
- **所有新增逻辑必须可被测试覆盖**
- 智能合约：关键函数必须有单元测试（Hardhat）
- 后端：核心业务逻辑必须有单元测试（Jest）
- 前端：关键用户流程建议有 E2E 测试

### 8. 命名规范严格遵守
- **所有变量名必须按照语言/框架规范**
- Solidity: `camelCase` 变量，`PascalCase` 合约/接口，`UPPER_CASE` 常量
- TypeScript: `camelCase` 变量/函数，`PascalCase` 类/接口/类型
- React: 组件 `PascalCase`，hooks `useCamelCase`
- 数据库：表名 `snake_case`，实体类 `PascalCase`

### 9. 依赖管理原则
- 禁止引入未经评审的第三方库
- 优先使用项目已有依赖，避免功能重复的包
- 新增依赖必须检查：活跃度、安全性、License、包大小
- 定期更新依赖以修复安全漏洞

## III. 技术栈约束

### Solidity (Smart Contracts)
- **版本锁定**：Solidity 0.8.20+（使用内置溢出检查）
- **标准库优先**：优先使用 OpenZeppelin Contracts
- **Gas 优化**：在不牺牲可读性前提下优化 gas 消耗
- **事件日志**：关键状态变更必须 emit Event

### Backend (NestJS + TypeScript)
- **框架规范**：严格遵循 NestJS 模块化架构
- **类型安全**：启用 `strict` 模式，禁止 `any` 类型（除特殊场景）
- **数据库访问**：使用 TypeORM Entity 和 Repository 模式
- **异步处理**：使用 async/await，避免回调地狱
- **配置管理**：使用 `@nestjs/config`，禁止硬编码配置

### Frontend (React + TypeScript)
- **状态管理**：简单状态用 useState/useContext，全局状态用 Zustand
- **数据请求**：使用 React Query 管理服务端状态
- **Web3 交互**：使用 Wagmi + Viem，避免直接操作 Web3.js
- **组件设计**：遵循单一职责，Props 明确类型定义
- **性能优化**：合理使用 useMemo/useCallback，避免过早优化

## IV. 开发流程规范

### 1. Spec-Driven Development (规范驱动开发)
- 任何新功能开发前必须先编写 `spec.md`
- Spec 必须包含：用户故事、功能需求、验收标准、安全考量

### 2. Plan Before Code (先设计后编码)
- 复杂功能必须先编写 `plan.md` 并进行"合宪性审查"
- Plan 必须包含：技术方案、数据结构、接口设计、风险评估

### 3. Test-Driven Development (测试驱动开发)
- 遵循 Red-Green-Refactor 循环
- 先写失败的测试，再实现功能，最后重构优化

### 4. Code Review Checklist
在任何代码审查时，必须检查：
- [ ] 是否违反 Constitution 任一条款
- [ ] 是否有未处理的错误
- [ ] 是否有未验证的输入
- [ ] 是否有安全漏洞（OWASP Top 10）
- [ ] 是否有测试覆盖
- [ ] 命名是否清晰
- [ ] 是否有必要的注释

## V. 禁止清单（AI 常犯错误）

### ❌ 绝对禁止
1. **智能合约中使用 `tx.origin` 进行授权**（必须用 `msg.sender`）
2. **前端存储或传输用户私钥**
3. **后端执行链上交易**（违反去中心化原则）
4. **在 `.env` 文件中存储真实私钥并提交到 Git**
5. **使用 `any` 类型逃避 TypeScript 检查**
6. **直接修改 node_modules 代码**
7. **在生产环境使用 `console.log` 暴露敏感信息**
8. **跳过输入验证直接操作数据库**

### ⚠️ 谨慎使用
1. 动态 SQL 拼接（易导致注入，优先用 ORM）
2. `eval()` 和 `new Function()`（严重安全风险）
3. 复杂的正则表达式（可能导致 ReDoS 攻击）
4. 大量使用 `!` 非空断言（可能隐藏潜在 bug）

---

## VI. 合宪性审查门禁

在以下阶段必须进行"合宪性审查"：

1. **设计阶段**：编写 `plan.md` 时，必须逐条对照本宪法自检
2. **实现阶段**：每个 PR 提交前，必须运行审查 Checklist
3. **Code Review**：审查者必须以本宪法为标准评审代码
4. **发布前**：重大版本发布前，必须进行全面合宪性审查

---

## VII. 例外处理流程

如遇特殊情况必须违反本宪法某条款：

1. 在代码中明确注释说明原因（`// CONSTITUTION EXCEPTION: 原因说明`）
2. 在 PR 描述中说明违反的条款和必要性
3. 获得至少 2 名 Core Contributor 的批准
4. 记录到项目决策日志
